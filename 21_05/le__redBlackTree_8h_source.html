    2Â  * @page c_redBlackTree Red/Black Tree API
    3Â  *
    4Â  * @ref le_redBlackTree.h "API Reference"
    5Â  *
    6Â  * <HR>
    7Â  *
    8Â  * A Red-Black Tree is a data structure representing a self-balancing binary search tree.
    9Â  * Tree consists of nodes maintaining links to the parent, left and right nodes. Advantage over a
   10Â  * linked-list is faster search based on the key comparison. Advantage over a hashtable is
   11Â  * simplified memory management (no additional allocation within the library), better
   12Â  * scalability up and down, and possibility to easily iterate the set in ascending/descending order.
   13Â  *
   14Â  * @section rbtree_createTree Creating and Initializing Red-Black Trees
   15Â  *
   16Â  * To create and initialize an RB Tree the user must create an le_rbtree_Tree_t typed object and
   17Â  * initialize it using le_rbtree_InitTree() routine. At this time the user has to provide a pointer
   18Â  * to the comparator function, that provides a way to perform a comparison between objects.
   19Â  * The tree <b>must</b> be initialized before it can be used.
   20Â  *
   21Â  * @code
   22Â  * // provide the comparator function
   23Â  * static int compare(const void* a, const void* b)
   24Â  * {
   25Â  *    // return negative, 0, or positive value
   26Â  * }
   27Â  *
   28Â  * // Create the tree.
   29Â  * le_rbtree_Tree_t MyTree;
   30Â  *
   31Â  * // Initialize the tree.
   32Â  * le_rbtree_InitTree(&MyTree, compare);
   33Â  * @endcode
   34Â  *
   35Â  * <b> Elements of @c le_rbtree_Tree_t MUST NOT be accessed directly by the user. </b>
   36Â  *
   37Â  *
   38Â  * @section rbtree_createNode Creating and Accessing Nodes
   39Â  *
   40Â  * Nodes can contain any data in any format and is defined and created by the user. The only
   41Â  * requirement for nodes is that it must contain an @c le_rbtree_Node_t link member. The link member
   42Â  * must be initialized by calling @c le_rbtree_InitNode() before it is added to the tree; at this
   43Â  * time, pointer to the key of this object must be provided.
   44Â  * The node can be added to the tree using the function @c le_rbtree_Insert().
   45Â  *
   46Â  * @code
   47Â  * // The node may be defined like this.
   48Â  * typedef struct
   49Â  * {
   50Â  *      MyKeyType key;
   51Â  *      dataType someUserData;
   52Â  *      ...
   53Â  *      le_rbtree_Node_t myLink;
   54Â  * }
   55Â  * MyNodeClass_t;
   56Â  *
   57Â  * void foo (void)
   58Â  * {
   59Â  *     // Create the node.  Get the memory from a memory pool previously created.
   60Â  *     MyNodeClass_t* myNodePtr = le_mem_ForceAlloc(MyNodePool);
   61Â  *
   62Â  *     // Initialize the node's link.
   63Â  *     le_rbtree_InitNode(&MyTree, &myNodePtr->key);
   64Â  *
   65Â  *     // Add the node to the tree by passing in the node's link.
   66Â  *     le_rbtree_Insert(&MyTree, &myNodePtr->myLink);
   67Â  * }
   68Â  * @endcode
   69Â  *
   70Â  * @section rbtree_find Finding node in a Tree
   71Â  *
   72Â  * To find the node in the tree by the given key, use @c le_rbtree_Find() function.
   73Â  * To obtain the object itself, use the @c CONTAINER_OF macro defined in
   74Â  * le_basics.h. Here's a code sample using @c CONTAINER_OF to obtain the node:
   75Â  *
   76Â  * @code
   77Â  * // declare and initialize the key
   78Â  * MyKeyType key = <...>;
   79Â  * // Assuming MyTree has been created and initialized and is not empty.
   80Â  * le_rbtree_Node_t* linkPtr = le_rbtree_Find(&MyTree, &key);
   81Â  *
   82Â  * // Now we have the link but still need the node to access user data.
   83Â  * // We use CONTAINER_OF to get a pointer to the node given the node's link.
   84Â  * if (linkPtr != NULL)
   85Â  * {
   86Â  *     MyNodeClass_t* myNodePtr = CONTAINER_OF(linkPtr, MyNodeClass_t, myLink);
   87Â  * }
   88Â  * @endcode
   89Â  *
   90Â  * The user is responsible for creating and freeing memory for all nodes; the RB Tree module
   91Â  * only manages the links in the nodes. The node must be removed from all trees before its
   92Â  * memory can be freed.
   93Â  *
   94Â  * <b>The elements of @c le_rbtree_Node_t MUST NOT be accessed directly by the user.</b>
   95Â  *
   96Â  * @section rbtree_traverse Traversing a Tree
   97Â  *
   98Â  * Tree can be traversed in an ascending or descending order (in the sense of greater/lesser
   99Â  * provided by the comparator function):
  100Â  *
  101Â  * @code
  102Â  * // Ascending order
  103Â  * le_rbtree_Node_t* linkPtr;
  104Â  * for (linkPtr = le_rbtree_GetFirst(&MyTree);
  105Â  *      linkPtr != NULL;
  106Â  *      linkPtr = le_rbtree_GetNext(&MyTree, linkPtr))
  107Â  * {
  108Â  *     MyNodeClass_t* myNodePtr = CONTAINER_OF(linkPtr, MyNodeClass_t, myLink);
  109Â  * }
  110Â  *
  111Â  * // Descending order
  112Â  * le_rbtree_Node_t* linkPtr;
  113Â  * for (linkPtr = le_rbtree_GetLast(&MyTree);
  114Â  *      linkPtr != NULL;
  115Â  *      linkPtr = le_rbtree_GetPrev(&MyTree, linkPtr))
  116Â  * {
  117Â  *     MyNodeClass_t* myNodePtr = CONTAINER_OF(linkPtr, MyNodeClass_t, myLink);
  118Â  * }
  119Â  * @endcode
  120Â  *
  121Â  * @section rbtree_remove Removing node from a Tree
  122Â  *
  123Â  * To remove a node from a Tree, use le_rbtree_RemoveByKey() function:
  124Â  * @code
  125Â  *     // Remove the node
  126Â  *     le_rbtree_RemoveByKey(&MyTree, &(myNodePtr->key));
  127Â  *     // Free the object
  128Â  *     le_mem_Release(myNodePtr);
  129Â  * @endcode
  130Â  *
  131Â  * or le_rbtree_Remove() function:
  132Â  * @code
  133Â  *     // Remove the node
  134Â  *     le_rbtree_Remove(&MyTree, &(myNodePtr->myLink));
  135Â  *     // Free the object
  136Â  *     le_mem_Release(myNodePtr);
  137Â  * @endcode
  138Â  *
  139Â  * @section rbtree_synch Thread Safety and Re-Entrancy
  140Â  *
  141Â  * All Red-Black Tree function calls are re-entrant and thread safe themselves, but if the nodes
  142Â  * and/or Tree object are shared by multiple threads, explicit steps must be taken to maintain
  143Â  * mutual exclusion of access. If you're accessing the same tree from multiple threads, you @e must
  144Â  * use a @ref c_mutex "mutex" or some other form of thread synchronization to ensure only one thread
  145Â  * accesses the tree at a time.
