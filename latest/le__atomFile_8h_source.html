   16Â  * An atomic file operation is an operation that cannot be partially performed. Either the entire
   17Â  * operation is performed or the operation fails. Any unclean reboot or power-cut should not lead to
   18Â  * corruption or inconsistency of the file. Also when a process is performing an atomic write on a

   24Â  * @c le_atomFile_Open() will block until it can obtain the lock. File must be closed using
   25Â  * @c le_atomFile_Close() or @c le_atomFile_Cancel() api. Both @c le_atomFile_Close() and
   26Â  * @c le_atomFile_Close() closes the file and releases the acquired resources. However,
   27Â  * @c le_atomFile_Close() transfers all changes to disk, @c le_atomFile_Cancel() no change is
   28Â  * reflected on file. A file can be deleted atomically using @c le_atomFile_Delete() api.
   29Â  *
   30Â  * For opening C standard file stream, please see @ref c_atomFile_streams section.
   31Â  *
   32Â  * Writing on the file descriptors obtained by this API same as writing to regular file descriptor.
   33Â  * That means, any write to a file descriptor using this API doesn't ensure that data is transferred
   34Â  * to disk. Data is only transferred to disk when le_atomFile_Close() returns successfully. This
   35Â  * behavior is same for file stream as well.
