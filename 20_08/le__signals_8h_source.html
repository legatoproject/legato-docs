    6Â  * <HR>
    7Â  *
    8Â  * Signals are software interrupts that can be sent to a running process or thread to
    9Â  * indicate exceptional situations. The action taken when an event is received depends on the
   10Â  * current settings for the signal and may be set to either:
   11Â  *
   12Â  *  - Operating systems default action.
   13Â  *  - Ignore the signal.
   14Â  *  - A custom handler.
   15Â  *
   16Â  * When a signal is received, unless it is ignored or blocked the action for the signal will preempt
   17Â  * any code that is currently executing.  Also, signals are asynchronous and may arrive at any time.
   18Â  * See http://man7.org/linux/man-pages/man7/signal.7.html for more details.
   19Â  *
   20Â  * The asynchronous and preemptive nature of signals can be difficult to deal with and is often a
   21Â  * source of race conditions.  Moreover asynchronous and preemptive signal handling is often
   22Â  * unnecessary so code often looks something like this:
   23Â  *
   24Â  * @code
   25Â  * // A global volatile atomic flag.
   26Â  * static volatile int GotSignal;
   27Â  *
   28Â  * void sigHandler(int sigNum)
   29Â  * {
   30Â  *      // Must only use asynch-signal-safe functions in this handler, see signals(7) man page for
   31Â  *      // more details.
   32Â  *
   33Â  *      // Set the flag.
   34Â  *      GotSignal = 1;
   35Â  * }
   36Â  *
   37Â  * int main (void)
   38Â  * {
   39Â  *      while(1)
   40Â  *      {
   41Â  *          // Do something.
   42Â  *          ...
   43Â  *
   44Â  *          if (GotSignal = 1)
   45Â  *          {
   46Â  *              // Clear the flag.
   47Â  *              GotSignal = 0;
   48Â  *
   49Â  *              // Process the signal.
   50Â  *              ...
   51Â  *          }
   52Â  *      }
   53Â  * }
   54Â  * @endcode
   55Â  *
   56Â  * In this code sample, the signal handler is only used to set a flag, while the main loop handles
   57Â  * the actual signal processing.  But handling signals this way requires the main loop to run
   58Â  * continuously.  This code is also prone to errors. For example, if the clearing of the flags was
   59Â  * done after processing of the signal, any signals received during processing of the signals will
   60Â  * be lost.
   61Â  *
   62Â  * @section c_signals_eventHandlers Signal Event Handlers
   63Â  *
   64Â  * The Legato signals API provides a simpler alternative, called signal events.  Signal events can
   65Â  * be used to receive and handle signals synchronously without the need for a sit-and-wait loop or
   66Â  * even a block-and-wait call.
   67Â  *
   68Â  * To use signal events, the desired signals must first be blocked using le_sig_Block() (see
   69Â  * @ref c_signals_blocking).  Then set a signal event handler for the desired signal using
   70Â  * @c le_sig_SetEventHandler().  Once a signal to the thread is received, the signal event handler
   71Â  * is called by the thread's Legato event loop (see @ref c_eventLoop for more details).  The handler
   72Â  * is called synchronously in the context of the thread that set the handler.  Be aware that if the
   73Â  * thread's event loop is not called or is blocked by some other code, the signal event handler will
   74Â  * also be blocked.
   75Â  *
   76Â  * Here is an example using signal events to handle the SIGCHLD signals:
   77Â  *
   78Â  * @code
   79Â  * // SIGCHILD event handler that will be called as a synchronous event.
   80Â  * static void SigChildEventHandler(int sigNum)
   81Â  * {
   82Â  *      // Handle SIGCHLD event.
   83Â  *      ...
   84Â  *
   85Â  *      // There is no need to limit ourselves to async-signal-safe functions because we are now in
   86Â  *      // a synchronous event handler.
   87Â  * }
   88Â  *
   89Â  *
   90Â  * COMPONENT_INIT
   91Â  * {
   92Â  *      // Block Signals that we are going to set event handlers for.
   93Â  *      le_sig_Block(SIGCHLD);
   94Â  *
   95Â  *      // Setup the signal event handler.
   96Â  *      le_sig_SetEventHandler(SIGCHILD, SigChildEventHandler);
   97Â  * }
   98Â  * @endcode
   99Â  *
  100Â  *
  101Â  * @section c_signals_mixedHandlers Mixing Asynchronous Signal Handlers with Synchronous Signal Event Handlers
  102Â  *
  103Â  * Signal events work well when dealing with signals synchronously, but when signals must be dealt
  104Â  * with in an asynchronously, traditional signal handlers are still preferred.  In fact, signal
  105Â  * event handlers are not allowed for certain signals like program error signals (ie. SIGFPE,
  106Â  * etc.) because they indicate a serious error in the program and all code outside of signal
  107Â  * handlers are considered unreliable.  This means that asynchronous signal handlers are the only
  108Â  * option when dealing with program error signals.
  109Â  *
  110Â  * Signal event handlers can be used in conjunction with asynchronous signal handlers but
  111Â  * only if they do not deal with the same signals.  In fact all signals that use signal events must
  112Â  * be blocked for every thread in the process.  The Legato framework takes care of this for you when
  113Â  * you set the signals you want to use in the Legato build system.
  114Â  *
  115Â  * If your code explicitly unblocks a signal where you currently have signal event handlers, the
  116Â  * signal event handlers will no longer be called until the signal is blocked again.
  117Â  *
  118Â  * @section c_signals_multiThread Multi-Threading Support
  119Â  *
  120Â  * In a multi-threaded system, signals can be sent to either the process or a specific thread.
  121Â  * Signals directed at a specific thread will be received by that thread; signals directed at
  122Â  * the process are received by one of the threads in the process that has a handler for the signal.
  123Â  *
  124Â  * It is unspecified which thread will actually receive the signal so it's recommended
  125Â  * to only have one signal event handler per signal.
  126Â  *
  127Â  * @section c_signals_limitations Limitations and Warnings
  128Â  *
  129Â  * A limitation of signals in general (not just with signal events) is called signal merging.
  130Â  * Signals that are received but not yet handled are said to be pending.  If another signal of the
  131Â  * same type is received while the first signal is pending, then the two signals will merge into a
  132Â  * single signal and there will be only one handler function call.  Consequently, it is not possible
  133Â  * to reliably know how many signals arrived.
  134Â  *
  135Â  *
  136Â  * @warning Signals are difficult to deal with in general because of their asynchronous nature and
  137Â  * although, Legato has simplified the situation with signal events certain limitations still exist.
  138Â  * If possible, avoid using them.
  139Â  *
  140Â  * @section c_signals_blocking Blocking signals
  141Â  *
  142Â  * Signals that are to be used with a signal event handlers must be blocked for the entire process.
  143Â  * To ensure this use le_sig_Block() to block signals in the process' first thread.  All other
  144Â  * threads will inherit the signal mask from the first thread.
  145Â  *
  146Â  * The example below shows how to use a signal event in a separate thread.
  147Â  *
  148Â  * @code
  149Â  * // SIGCHILD event handler that will be called as a synchronous event in the context of the
  150Â  * // workThread.
  151Â  * static void SigChildEventHandler(int sigNum)
  152Â  * {
  153Â  *      // Handle SIGCHILD event.
  154Â  *      ...
  155Â  *
  156Â  *      // There is no need to limit ourselves to async-signal-safe functions because we are now in
  157Â  *      // a synchronous event handler.
  158Â  * }
  159Â  *
  160Â  *
  161Â  * // Work thread's main function.
  162Â  * static void* WorkThreadMain(void* context)
  163Â  * {
  164Â  *      // Setup the signal event handler.
  165Â  *      le_sig_SetEventHandler(SIGCHILD, SigChildEventHandler);
  166Â  *
  167Â  *      // Start this thread's event loop.
  168Â  *      le_event_RunLoop();
  169Â  *
  170Â  *      return NULL;
  171Â  * }
  172Â  *
  173Â  *
  174Â  * // Main thread code.
  175Â  * COMPONENT_INIT
  176Â  * {
  177Â  *      // Block Signals that we are going to set event handlers for in the main thread so that all
  178Â  *      // subsequent threads will inherit the same signal mask.
