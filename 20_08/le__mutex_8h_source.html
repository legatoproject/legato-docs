    8Â  * The Mutex API provides standard mutex functionality with added diagnostics capabilities.
    9Â  * These mutexes can be shared by threads within the same process, but can't
   10Â  * be shared by threads in different processes.
   11Â  *
   12Â  * @warning  Multithreaded programming is an advanced subject with many pitfalls.
   13Â  * A general discussion of why and how mutexes are used in multithreaded programming is beyond
   14Â  * the scope of this documentation.  If you are not familiar with these concepts @e please seek
   15Â  * out training and mentorship before attempting to work on multithreaded production code.
   16Â  *
   17Â  * Two kinds of mutex are supported by Legato:
   18Â  *  - @b Recursive or
   19Â  *  - @b Non-Recursive
   20Â  *
   21Â  * All mutexes can be locked and unlocked. The same lock, unlock, and delete
   22Â  * functions work for all the mutexes, regardless of what type they are.
   23Â  *
   24Â  * A recursive mutex can be locked again by the same thread that already has the lock, but
   25Â  * a non-recursive mutex can only be locked once before being unlocked.
   26Â  *
   27Â  * If a thread grabs a non-recursive mutex lock and then tries to grab that same lock again, a
   28Â  * deadlock occurs.  Legato's non-recursive mutexes will detect this deadlock, log a fatal error
   29Â  * and terminate the process.
   30Â  *
   31Â  * If a thread grabs a recursive mutex, and then the same thread grabs the same lock again, the
   32Â  * mutex's "lock count" is incremented.  When the thread unlocks that mutex, the lock count is
   33Â  * decremented.  Only when the lock count reaches zero will the mutex actually unlock.
   34Â  *
   35Â  * There's a limit to the number of times the same recursive mutex can be locked by the same
   36Â  * thread without ever unlocking it, but that limit is so high (at least 2 billion), if that
   37Â  * much recursion is going on, there are other, more serious problems with the program.
   38Â  *
   39Â  * @section c_mutex_create Creating a Mutex
   40Â  *
   41Â  * In Legato, mutexes are dynamically allocated objects.  Functions that create them
   42Â  * return references to them (of type le_mutex_Ref_t).
   43Â  *
   44Â  * Functions for creating mutexes:
   45Â  *  - @c le_mutex_CreateRecursive() - creates a recursive mutex.
   46Â  *  - @c le_mutex_CreateNonRecursive() - creates a non-recursive mutex.
   47Â  *
   48Â  * All mutexes have names, required for diagnostic purposes.  See
   49Â  * @ref c_mutex_diagnostics below.
   50Â  *
   51Â  * @section c_mutex_locking Using a Mutex
   52Â  *
   53Â  * Functions for locking and unlocking mutexes:
   54Â  *  - @c le_mutex_Lock()
   55Â  *  - @c le_mutex_Unlock()
   56Â  *  - @c le_mutex_TryLock()
   57Â  *
   58Â  * It doesn't matter what type of mutex you are using, you
   59Â  * still use the same functions for locking and unlocking your mutex.
   60Â  *
   61Â  * @section c_mutex_delete Deleting a Mutex
   62Â  *
   63Â  * When you are finished with a mutex, you must delete it by calling le_mutex_Delete().
   64Â  *
   65Â  * There must not be anyone using the mutex when it is deleted (i.e., no one can be holding it).
   66Â  *
   67Â  * @section c_mutex_diagnostics Diagnostics
   68Â  *
   69Â  * The command-line diagnostic tool @ref toolsTarget_inspect can be used to list the mutexes
