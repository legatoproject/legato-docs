    8Â  * Unit testing is an important aspect of a quantifiable quality assurance methodology.  Although
    9Â  * unit testing requires some extra overhead (the writing of the unit tests) during the development
   10Â  * process it can provide enormous benefits during the project life cycle.
   11Â  *
   12Â  * One benefit of writing unit tests is that it gets the developer using the interface to the
   13Â  * unit they designed. This forces the developer to think about, and hopefully design for,
   14Â  * usability of the interface early in the development cycle.
   15Â  *
   16Â  * Another major benefit to unit testing is that it provides a documented and verifiable level of
   17Â  * correctness for the designed unit. This allows the developer to refactor the code more
   18Â  * aggressively and to quickly verify its correctness.  Unit tests can also be used to perform
   19Â  * regression testing when adding new features.
   20Â  *
   21Â  * Despite the benefits of unit testing, unit tests are often omitted because of the initial
   22Â  * overhead of writing the tests and the complexity of testing frameworks.  Legato's Unit Test
   23Â  * Framework is simple to use and very flexible and lightweight consisting of some
   24Â  * handy macros.
   25Â  *
   26Â  * The Legato test framework outputs test results to the log in TAP format (prefixed by "TAP |")
   27Â  * This allows test results to be processed with many test harnasses.
   28Â  *
   29Â  * @section c_test_setup Setting Up the Test Framework
   30Â  *
   31Â  * To setup the Legato Test Framework, call the @c LE_TEST_PLAN macro, once before any tests
   32Â  * are started. The macro takes the total number of planned tests as single argument.
   33Â  *
   34Â  * @section c_test_testing Performing Tests
   35Â  *
   36Â  * To perform tests, call the @c LE_TEST_OK macro.  The first argument is whether the
   37Â  * test passed (true or false).  The second argument is the name of the test.
   38Â  *
   39Â  * For example:
   40Â  *
   41Â  * @code
   42Â  * #include "legato.h"
   43Â  *
   44Â  * // Returns true if the test passes, otherwise returns false.
   45Â  * bool ComplexTest(void)
   46Â  * {
   47Â  *     int expectedValue;
   48Â  *
   49Â  *     // Do some initializations and/or calculations.
   50Â  *     ...
   51Â  *
   52Â  *     // Call one of the unit-under-test's interface function and check it's return value against
   53Â  *     // an expected value that was calculated earlier.
   54Â  *     return (unitUnderTest_foo2() == expectedValue);
   55Â  * }
   56Â  *
   57Â  *
   58Â  * int main (void)
   59Â  * {
   60Â  *     // Setup the Legato Test Framework.
   61Â  *     LE_TEST_PLAN(2);
   62Â  *
   63Â  *     // Run the tests.
   64Â  *     // Do some initializations and/or calculations.
   65Â  *     ...
   66Â  *
   67Â  *     LE_TEST_OK(TestFunction(arguments) == EXPECTED_VALUE, "simple test");
   68Â  *     LE_TEST_OK(ComplexTest(), "complex test");
   69Â  *
   70Â  *     // Exit with the number of failed tests as the exit code.
   71Â  *     LE_TEST_EXIT;
   72Â  * }
   73Â  * @endcode
   74Â  *
   75Â  * @section c_test_exit Exiting a Test Program
   76Â  *
   77Â  * When a test program is finished executing tests and needs to exit, it should exit
   78Â  * using the LE_TEST_EXIT macro.
   79Â  *
   80Â  * If a test suite needs to exit early it should use the LE_TEST_FATAL() macro.  This will
   81Â  * log a message indicating the test suite has been aborted.  LE_ASSERT()
   82Â  * and LE_FATAL() macros should not be used as they will not print this message.
   83Â  *
   84Â  * As a convenience you can also use the LE_TEST_ASSERT() macro which will abort the test suite
